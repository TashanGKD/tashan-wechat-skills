# 团队协作（团队协作记录）

本项目用仓库里的 `团队协作记录/` 文件夹作为**异步协作频道**。核心理念一句话：

> **文件即消息。** 每个话题一个 Markdown 文件，所有人（人 + 各自的 AI 替身）在文件里你来我往；改动通过 git commit / pull 同步。它同时是**沟通区**和**决策档案**。

为什么这么设计：多人 + AI 一起做项目，靠零散的微信/口头沟通会**丢话、决策没法追溯、新人接不上**。把沟通落到 git 里，谁、什么时候、说了什么、定了什么，一清二楚，且新成员读一遍就能上手。

你作为某位成员的 AI 替身（最常见是代表 `[Alice]`），在这套系统里的发言**视同你所代表的成员本人**。所以要像他本人一样，遵守下面的地基规范，并按「路由」去读对应详规。

---

## 🧭 路由：按你要做的事，去读对应详规

本 SKILL.md 只放**所有协作都适用的地基**（理念 / 先读现场 / 署名 / 找谁负责 / git 纪律）。具体怎么做，按下表跳读——**别凭记忆，先读对应文件再动手**：

| 你要做的事 | 读这个 |
|---|---|
| **发帖 / 回帖 / 收敛结论**（话题→帖→回复 模板、append-only、状态标签、附件、好回帖示范、🚩红线自检、格式 gate） | [`references/posting.md`](./references/posting.md) |
| **收尾 / 归档会话 / 记忆层**（记忆层 = 本地会话的**去重分支树镜像**、一条命令 `build_session_tree.py` 重生成「对话树」、每节点配研究历程/动机、Mermaid 思维导图） | [`references/worklog.md`](./references/worklog.md) |
| **读 / 查 / 找本地历史对话记录**（"帮我找那个会话 / 我以前在哪做过 X / 回顾之前的对话 / 上次那个网页是哪个会话做的"）——**先刷新对话树、再在工作日志里检索、命中后按真源指针下钻**，别裸 grep jsonl（见下「地基五」） | [`references/worklog.md`](./references/worklog.md)（§读协议） |
| **把对话转成「完整对话记录」统一格式**（格式定义、转化要求、脱敏规则、给其他框架写适配器；头部含指向原始 jsonl 的**真源指针**） | [`references/conversation-log-spec.md`](./references/conversation-log-spec.md)（脚本在 [`scripts/`](./scripts/)） |
| **生成对话「续接包」/ 换框架换账号继续某条对话**（把任意一条对话**从根整条** compact 成即开即用的交接包，供在新会话/新框架/新账号里粘贴续接——`pack_conversation.py` 打包 → 派子智能体用 compaction 模板总结 → 套续接壳。**定位要续的那条**有两入口：A 看思维画布点复制节点起点的 session-id；B 用户说"回顾项目工作找 xx 相关的对话"→ `find_conversations.py` 出表格〔session-id｜总结｜最近3个完整交互〕） | [`references/conversation-handoff.md`](./references/conversation-handoff.md)（脚本 `scripts/pack_conversation.py`、`scripts/find_conversations.py`） |

> **在任何新仓库首次落地本协作法**（一次性 bootstrap）：① 该项目须是 git 仓库、且有 `CONTRIBUTING.md`（git 规范）；② 建 `团队协作记录/README.md`（沟通范式 + 话题索引表）与 `团队协作记录/画像/README.md`（「找谁负责」速查表 + 成员画像）；③ 启用格式 gate：`git config core.hooksPath .githooks`（gate 由本 skill `scripts/` 下的 `check_posts.py`/`check_entrypoints.py`/`check_pii.py` 校验，hook 接法见 `references/posting.md`）。之后按下表日常使用——帖子格式会被自动校验，不合模板会被拦下。
>
> 本 skill 是**仓库无关**的：把它放进任意项目的 `.claude/skills/`（或 `.cursor/skills/`），完成上面 bootstrap 即可用；正文里说的"本项目/本仓库"都指你当前所在的那个仓库。

## 地基一：先读现场，别凭记忆

本 skill 讲的是**方法和规范**；**当前的真实状态**永远以仓库里的活文件为准。动手前先读：

1. `团队协作记录/README.md` —— 沟通范式 + 玩法规矩 + **话题索引表**（有哪些帖、各自状态）
2. `团队协作记录/画像/README.md` —— **「找谁负责」速查表** + 成员索引
3. 你要回复或相关的那个话题帖本身

读完你才知道：现在有哪些议题、轮到谁、哪些已决、这事归谁。

---

## 地基二：署名规范

- 你代表谁，就署谁的名，格式 `[名字]`，例如 `[Alice]`、`[XX78]`。
- **AI 替身视同本人**：你代 Alice 写，就署 `[Alice]`，不要另署 "Claude" 或 "AI"——对方不关心是人还是 AI 写的，只关心这是谁的意见。
- 发起新话题前，必须先明确**发起人是谁**：这帖是代表 `[Alice]` 发起、代表 `[XX78]` 发起，还是你以当前操作者身份发起。文件头的 `发起人`、讨论区第一条署名、索引表里的 `发起` 必须一致。
- 不确定自己代表谁、谁是发起人，或被要求替某人发言但越界，先问，别擅自代别人发帖或拍板。
- 署名同样用于工作日志（`--person <你代表谁>`）：识别"谁"靠声明身份。

---

## 地基三：找谁负责——路由问题、别越界拍板

`团队协作记录/画像/` 里每人一张画像卡，外加一张「找谁负责」速查表。用法：

- **遇到问题，先查速查表**，知道该 @ 谁，别问错人。
- **review ≠ decide**：请人帮忙检查、提意见，任何人都能做、且越多人看越好；但**最终拍板权按画像走**。例如商务/合同/范围归 `[Alice]`，技术实现/架构归 `[XX78]`。请对方 review 你域内的东西，是要他的视角，不是把决定权交出去。
- **灰区（落在两人之间、或范围 ↔ 实现冲突）不归任何单人 → 直接开一个话题帖讨论。**

发帖时也据此自检：**我是在让对方"提意见"，还是在让对方"替我拍板我自己该决定的事"？** 后者是错的——改成：我陈述我的决定（知会），只把对方域内的事抛给对方。

## 地基四：这是个 git 仓库，守工程纪律

`团队协作记录/` 活在一个多人共享的 git 仓库里，所以：

- **先看再改**：动任何已存在的文件前先读它；不要凭"它应该是空的/应该是什么样"就覆盖。仓库里别人随时在推东西。
- **遵守团队的 git 规范**：见仓库根的 `CONTRIBUTING.md`（分支模型、Conventional Commits、保护 main、PR 流程）。
- **推送被拒就 `git fetch` + `git rebase origin/main` 再推**，绝不 force push、绝不覆盖别人的提交。先确认对方的提交和你的没有文件重叠，再 rebase。
- 提交信息用 Conventional Commits，协作记录的改动用 `docs(collab): …`。

## 地基五：读本地历史对话记录——先更新工作日志、以它检索、按真源下钻

当任务要**读 / 查 / 找本地过去的对话记录**（"我以前在哪做过 X""上次那个网页/脚本是哪个会话做的""回顾之前的讨论"），别一上来就裸 `grep ~/.claude/projects/**.jsonl`。本地 jsonl 是**高度重复、还在增长**的一堆文件——直接扫它，① 容易拿到一份**过期快照**、漏掉刚开的新会话（本项目就真踩过：一次早扫只见 16 个会话，实际已有 24 个，今天的并行会话全漏了）；② 满是续接重抄的重复，噪声大。正确姿势是把**去重后的对话树当检索索引**，需要逐字时再回**真源**：

1. **先更新**（只在"明知有很新的对话要找"时；日常检索可跳过）——把对话树+记忆索引刷到最新：
   ```bash
   PYTHONIOENCODING=utf-8 python3 .claude/skills/team-collab/scripts/build_session_tree.py --person <你代表谁> --if-stale
   PYTHONIOENCODING=utf-8 python3 .claude/skills/team-collab/scripts/build_memory_index.py --person <你代表谁>   # 增量重嵌新/变节点
   ```
   `--if-stale` 秒判新鲜度、陈旧才重建、否则秒过；要稳就去掉它做无条件全量重建。
2. **在工作日志里检索（优先语义召回）**——首选 **`recall-memory` skill / `query_memory.py "<问题>" --person <你代表谁>`**：直接按意思命中相关节点，每条带**真源指针**。要人工浏览再看对话树里的 `目录.md`（可跳转任务总目录）、`思维导图.md`、命中节点的 `段.md`。总之**以工作日志/记忆向量库为入口，不是裸 grep jsonl**。
3. **命中后按真源下钻**——若命中节点的 **真源指针**（`段.md` 头部 `真源(source-of-truth):` 行 / `tree.json` 的 `source_files`）指向的原始 `.jsonl` **就在本机、可读**（即你就是这份工作日志的所有者）→ 顺着它回原始 jsonl 读**无损细节**（`段.md` 已够时就不必下钻）；若那是**别人的**节点、源不在你机上 → 就以 `段.md` 为准（读不到别人的真源是正常的）。

一句话：**工作日志（对话树）= 检索索引（先刷新再查）；原始 jsonl = 唯一真相（自己拥有时才下钻）。** 两者靠 `段.md` 头部的真源指针连起来。详规见 [`references/worklog.md`](./references/worklog.md) 的「读协议」。

## 小结：接手本项目的智能体，按这个顺序做

1. 读 `团队协作记录/README.md` 和 `画像/README.md`，搞清现状、范式、找谁负责（地基一）。
2. 明确你代表谁；发起新话题前先明确发起人是谁，发言就署谁（地基二）。
3. 要沟通就在对应**话题**（一个 md）下**发帖 / 回帖**——**先说想做什么、再说方案**，一件能独立收敛的事一个帖。详规与模板读 [`references/posting.md`](./references/posting.md)，照做。
4. 只在帖的「💬 回复」区追加、不改别人的话；推进到结论就写 `✅ 结论` 并改状态（提交会过格式 gate）。
5. 路由按「找谁负责」，请人 review 而非替你拍板，灰区开新帖（地基三）。
6. 守 git 纪律：先看再改、Conventional Commits、冲突就 rebase（地基四）。
7. 收尾时按 [`references/worklog.md`](./references/worklog.md)：用 `/worklog`（或 `scripts/build_session_tree.py`）**重生成「对话树」**——本地 Claude Code 会话的去重分支树镜像，随对话增多自动长大；再补当前节点的 `研究历程.md`/`动机日志.md` 实质内容、脱敏过目、**精确 `git add` 本人 `对话树/`**（别 `-A`）提交。
