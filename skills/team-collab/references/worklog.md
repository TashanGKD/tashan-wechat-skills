# 记忆层 / 对话树详规（收尾归档）

> 这是 `team-collab` 的**记忆层规范**。SKILL.md 给地基与路由；具体怎么收尾、把会话沉淀进团队记忆，看这里。

## 读协议：要读本地历史对话，先更新工作日志→以它检索→按真源下钻

本文件多数篇幅在讲**怎么建**对话树（收尾归档）。但对话树同样是**读**的入口——当有人要"翻本地过去的对话记录"（"我以前在哪做过 X""上次那个网页/脚本/相图是哪个会话做的""回顾之前的讨论定了什么"），走这套、别裸 `grep ~/.claude/projects/**.jsonl`：

**为什么不能裸 grep 原始 jsonl。** 本地 jsonl 有两个"病"（详见下节「本地是怎么存的」）：① **续接即重抄前文**，同一段对话在本地躺 N 份、噪声极大；② 它**一直在长**，你某一刻扫到的是**快照**，容易漏掉刚开/还在写的会话。真实教训：本项目一次早扫只见到 16 个会话就当"全部"，实际已有 24 个——今天的并行会话全漏了。对话树把这堆去重成一棵干净、可检索的树，正是为读而生。

**三步：**

1. **先更新**（只在要找"很新的对话"时；日常检索可跳过以求快）：
   ```bash
   PYTHONIOENCODING=utf-8 python3 .claude/skills/team-collab/scripts/build_session_tree.py --person <你代表谁> --if-stale
   PYTHONIOENCODING=utf-8 python3 .claude/skills/team-collab/scripts/build_memory_index.py --person <你代表谁>   # 增量重嵌新/变节点
   ```
   - `--if-stale`：秒级新鲜度检查（`stat` 源会话 mtime + 有无新 session vs 现有 `tree.json`），**陈旧才重建、否则秒过**。不加 = **无条件全量重建**（最稳，别信快照）。
   - `build_memory_index.py` 增量 upsert（按内容 hash 只重嵌变化/新增节点），让语义检索跟上最新的树。
   - （Windows 记得 `PYTHONIOENCODING=utf-8`；脚本会自动探测装了 chromadb 的解释器（python3/python/py 等），不必是当前解释器。verify_tree 要板上钉钉就**另跑一次**看退出码、别接管道。）
2. **检索（优先语义召回）**，入口是**记忆向量库/对话树**、不是裸 jsonl：
   - **首选语义**：`recall-memory` skill 或 `python3 .claude/skills/team-collab/scripts/query_memory.py "<问题>" --person <你代表谁> --json`——按意思命中相关节点，每条回带 `摘要/snippet/session/真源(source_files)/段.md/锚点`。多数问题一步到位。
   - **要人工浏览**：对话树里的 `目录.md`（全项目任务总目录，叶子 `↪` 跳到对话对应行）、`思维导图.md`（Mermaid 全局概览）、`TREE.md`/`tree.json`（`tree.json` 每节点带 `sessions`/`uuids`/`source_files`）、命中节点的 `段.md`（去重后的对话原文、已脱敏）。
3. **命中后按真源下钻**（要逐字/要工具入参与结果的细节时）：
   - 每个 `段.md` 头部有 `真源(source-of-truth):` 行（`tree.json` 里对应 `source_files`），指向该节点记录所属的**本机原始 `.jsonl` 绝对路径**，`节点段:` 行还附了 `uuid` 首末。
   - **若真源在本机、可读**（即**你就是这份工作日志的所有者**）→ 顺着它回原始 jsonl 按 `uuid` 精确定位、读无损细节。
   - **若那是别人的节点、源不在你机上** → 就以 `段.md` 为准（读不到别人的真源是正常的——这正是把真源指针写进 md 的意义：能下钻的人下钻，不能的人有 md 兜底）。

一句话：**对话树 = 检索索引（先刷新再查）；原始 jsonl = 唯一真相（自己拥有才下钻）。** 真源指针（Point：`段.md` 头部）是二者之间的回程票。

## 理念：不自创组织，镜像 Claude Code 原生会话 → 去重分支树

记忆层 = **凡"够到本项目"的全部 Claude Code 会话的去重分支树镜像**：
- **范围 = 项目，不是文件夹**。一个会话只要够到本项目（用到本 skill / 引用了本仓库目录名），就属于本项目——**不论它从哪个 cwd 启动、存在哪个 `~/.claude/projects/<*>/` 目录下**。同一个仓库常被从多个目录启动过（仓库本身 / 它的上级目录），对话因此被散存进多个项目目录；记忆层要把它们**全收齐**，而不是只认某一个目录。判定靠"会话内容是否引用本仓库目录名"（ASCII 标记，可直接 grep）。
- **照搬 CC 原生组织**（每个会话一个 `<session-id>.jsonl` + 同名子目录装子智能体），不自己发明命名/锚定规则。
- 只做两件**必要处理**：① 每段对话 `jsonl → 统一 md`（忠实改格式、不概要）；② **脱敏**。
- 按 `uuid`/`parentUuid` 拼成**树**、按 `uuid` **去重**：共享主干只存一份，分支只存分叉后的增量。**文件夹/cwd 不进结构**——树里只有"对话（并列）+ 分支（子会话）"。
- 每个节点旁配 `研究历程.md` / `动机日志.md` = **该节点的"记忆"**（人写）。

为什么这么做：session-id 在一段对话里会反复变（压缩续接 / 恢复 / 分支各换新 id），我们曾自创"按主题文件夹 + session 锚定 + 新 sid 新日志"那套规则，越绕越乱、还重复。直接镜像 CC 原生结构成树——分支、续接、子智能体全自然落位，最干净统一。（旧的"按主题五件套"已归档在 `_旧归档/`。）

> **唯一机制 = 全量重建。** 没有"手动判这次会话是新是旧 / 它从哪分支 / 只同步我这一段"这回事——**这些全是 `build_session_tree.py` 内部按 `uuid`/`parentUuid` 自动做的**：你只管重建整棵树，它自己把新会话落成新根/新分支、把旧会话续写回原节点、共享主干按 uuid 去重。**"索引"就是它生成的 `tree.json`**（含每节点的 session 与 uuid 序列）；**不存在、也不需要**单独的"增量同步脚本"或 INDEX 文件。重建是无损 + 崩溃安全 + 可校验的，所以"重建一切"既简单又可靠，不要去手工拆"增量"。

## 本地是怎么存的 · 为什么必须去重（对话树存在的根本原因）

要懂对话树为什么这么设计，得先懂 Claude Code **本地**怎么存会话——对话树就是冲着它的两个特性来的。**这套原理写在这里、不只藏在脚本里**：脚本会变、原理不变，照着原理才知道脚本在干嘛、出问题怎么判。

**会话存哪**：每个会话一个 `~/.claude/projects/<编码cwd>/<session-id>.jsonl`，每行一条记录，带 `uuid`（本条的唯一 id）+ `parentUuid`（上一条的 id）——靠这对 id 把记录串成链 / 树。

**"开分支"有两种，存法完全不同（关键）：**

1. **同一会话内：编辑重发 / 回到上面某条重问**（session-id **不变**）
   → 还在**同一个 jsonl 文件里**，靠 `parentUuid` 长成一棵**树**（一个 parent 挂多个 child），旧分支也保留。
   → **不产生重复**：主干记录在文件里只存一份，分支只是多挂几条。
   （实测：单个 `5949998f.jsonl` 内有 13 个分叉点、`c3b09cca.jsonl` 有 32 个，记录并不翻倍。）

2. **跨会话：上下文满了压缩续接 / 恢复历史 → session-id 变了**（即"session1 → session1-2"）
   → 生成**一个新的 jsonl 文件**，它把**开分支前的记录整段原样拷过来（uuid 完全相同）+ 接上新内容**。
   → 即 **`session1-2` 文件 = `session1` 的全部记录（逐条拷贝）+ 续接后的增量**。不是新编号、也不是摘要，是**逐条同 uuid 的复制**。
   （实测续接链：`879a3e89` 639 条 → `a988c98f` 677 条【含 879a3e89 的 633 条·99% 原样拷贝 + 44 条新】→ `c3b09cca` 2090 条【含 a988c98f 的 674 条·99% + 更多】。）

**后果（本地的"病"）**：**每续接一次，前文就被物理重抄一份。** 一段长对话续接 N 次，最早那批记录就在本地躺 N+ 份。（实测：本会话最早的核心 243 条，同时躺在 `5949998f / ef9bcfc8 / dd9db317 / 879a3e89 / a988c98f / c3b09cca` 六个文件里、份份相同。）把这些 jsonl 原样堆上 GitHub = 同一段对话重复 N 遍、还乱。

**这就是对话树要解决的根本问题，也是"按 uuid 去重"的全部理由：**
- 拷贝时 **uuid 完全一样** → 同一条记录无论出现在几个文件里都是**同一个 uuid** → 对话树**按 uuid 认成同一个节点、只存一份**（作为共享主干 / 祖先节点）。
- 每个续接 / 分支节点**只存"分叉之后的增量"**，不再重复前缀。
- 所以：**本地 = 高度重复的一堆 jsonl；对话树 = 去重后的同一棵树**（共享主干一份、分支只存增量）。内容一条不少（`verify_tree` 保证能从树无损重构出每条真实会话链），但重复被压掉、形状变清楚。

> 一句话：**Claude Code 本地"续接即重抄前文（uuid 不变）"，越用越重复；对话树按 uuid 去重，把这堆重复还原成一棵干净的树。** 这是整个记忆层设计的出发点——也是为什么我们不直接传 jsonl、而要建对话树。

## 收尾：一条命令重生成对话树（树会自动长大）

```bash
# ① 重生成整棵树（扫本地所有会话 → 去重树 → A 布局 + tree.json + TREE.md + 思维导图.md + 规范.md）
python3 .claude/skills/team-collab/scripts/build_session_tree.py --person <你代表谁>
```

它做：扫 `~/.claude/projects/` 下**所有引用了本仓库的会话**（跨项目目录/cwd，自动去重合并）→ 拼去重树 → 落 A 布局节点目录（每节点生成 `段.md`，脱敏）→ 脚手架缺失的 `研究历程.md`/`动机日志.md`（**不覆盖**已写的）→ 生成 `tree.json`/`TREE.md`/`思维导图.md`/`规范.md` → 打印脱敏报告。**只写本地、不提交。**（建树前可先 `--list` 过目"判定为属于本项目"的会话清单，防误收无关项目。）

**重建是无损的**：重生成前按"段身份（段首 `### [时间]` 时间戳，与路径/别名/记录数无关）"快照所有已填的 `研究历程`/`动机`（两者各自独立保留），`rmtree` 清掉旧树消除孤儿，再据段身份贴回——所以放心重建，改别名/源会话增减/已填内容都不会丢、不留孤儿目录。**纯续接/工具桥接段**（无用户实质文本）由脚本**自动写成"空桥接节点"完整内容、不算待补**，无需人工补。

**默认收全（不要轻易用 manifest 排除）**：默认 auto-discover 收齐所有引用本仓库的会话——这是对的，宁可全收。曾经"反复重建越来越乱"是 builder 的**段排序 bug**（已修：段按 parentUuid 因果链、不再按时间戳排），不是会话太多；**别用 `--manifest` 去"固定"集合来回避它**——那会把真实会话（如从仓库子目录跑的合同续接段）**漏在树外**，正是要防的"遗漏"。`--manifest <清单>` 只在你**确认**某些顶层会话是无关项目/纯工具噪声、要排除时才用（清单放 person 目录、别放 `对话树/` 里以免被 rmtree）；用了就同一份传给 `verify_tree.py`。

**重建后务必跑完整性校验**（证明"拼起来=原始会话、不丢不错配"）：
```bash
# ② 校验：节点不重叠 / 无用户内容遗漏 / 每段是真实连续 parentUuid 链 / 根→叶可无损重构真实会话
python3 .claude/skills/team-collab/scripts/verify_tree.py --person <你代表谁>
```
verify 把结论分成两类，**别混为一谈**：
- 🔴 **结构性问题**（重复/幽灵/断裂/接缝/根错/重构不符/段.md 数不符）= 树**自身不自洽**，是真错配，**退出码 1、必须修**。正常流程下**先 build 再 verify、且对同一次 build 的产物跑**——刚重建出的树结构性应为 0；若仍 🔴，多半是 builder/源有真问题，别提交。（`build_session_tree.py` 建完已**自动跑一次自检**，红了会警告。）
- 🟡 **陈旧**（源里有、树里没有的新会话/增长）= 不是错，就是"**该重建了**"，**退出码 0、只提醒**。`--strict` 时才把陈旧也算失败。

**查退出码别接管道**：`verify_tree.py | tail` 会把退出码换成 `tail` 的——要判健康直接看脚本退出码或它打印的「🔴 结构性问题 N」。

**提交受闸门保护**：pre-commit 暂存 `对话树/` 时会自动跑 **完整性硬门**（只拦 🔴 结构性、🟡 陈旧不拦）+ **PII 闸**（`check_pii.py` 扫漏脱敏的手机/邮箱/微信号/长号）。所以"红着的树"或"带 PII 的树"提交会被拦。

`剪枝` 那部分是"无任何用户输入的死支"的 agent/工具记录（要连死支也留：建树加 `--keep-stubs`）。

**命令跑完，两件人工活（脚本替不了）：**
1. **补当前节点的 `研究历程.md` / `动机日志.md` 实质内容**——脚本只给骨架 + `⚠️ 待补`。留模板不填 = 没做。（判"已填"用行锚定占位符，所以正文里**可以**出现"待补"二字、不会被误判。）
2. **精确提交**——看脱敏报告，再**只 `git add` 本人这一棵树**：`git add 团队协作记录/智能体工作日志/<你>/对话树/`（**绝不 `git add -A`**；多 person 时也只 add 自己那个 `<你>/对话树/`、不碰别人的）→ `docs(collab): 更新对话树` → 冲突就 fetch+rebase、不强推。提交走 **main 直推**即可（协作记录/记忆层是文档类、非产品代码，不需开 PR；CONTRIBUTING 的 PR 流程针对 `business/` 产品代码）。

**两条边界（skill 明确）：**
- **源会话被删/归档**：树据**现存**源会话重拼——某会话的 `.jsonl` 被物理删了，它的节点（连同人写的研究历程/动机）**会随之从树里消失**。要保留就**别删源**，或先把该节点目录手动挪进 `_旧归档/`（builder 不碰 `_旧归档/`）再删源。
- **`--with-subagents` 默认不开**：默认不把子智能体镜像进树（子智能体 transcript 在 `<sid>/subagents/` 下、不算"对话"）。本会话子智能体产物值得留存时才显式加该 flag。

> Claude Code 用户用 `/worklog` 一键走这套。其他工具（Codex/Cursor）见下「跨框架」。

## 结构（A 布局：目录即树，物理嵌套）

```
团队协作记录/智能体工作日志/<人>/对话树/
  对话N（sid前8）/          ← 一段独立对话（根，按时间排）
    段.md                   ← 本节点对话（统一格式·脱敏·不概要）  【生成物·勿手改】
    研究历程.md              ← 怎么探索/转向/踩坑                 【人写·脚本不覆盖】
    动机日志.md              ← 每步为什么/否决了什么              【人写·脚本不覆盖】
    子智能体/                ← 本段 spawn 的子智能体（若有）
    分支M（sid前8）/         ← 真实分叉后的增量（递归同构）
  TREE.md  思维导图.md  tree.json  规范.md   ← 均为生成物·勿手改
```

- **节点 = 一段无分叉对话**；**分叉点 = 真实分支**。两类分支都如实落树（见上「本地是怎么存的」）：① **同会话内编辑重发 / 回退**——session-id **不变**、本就在同一文件内成树；② **压缩续接 / 恢复**——**换新 session-id**、新文件拷贝前缀。builder 按 `uuid`/`parentUuid` 对两类一视同仁地拼、去重。
- **去重**：同 uuid = 同一节点，共享主干只存一份。
- **命名**：`对话N` / `分支M`（时间序）+ 括注 session-id 前 8 位；层级靠物理嵌套。
- **唯一标识 = session-id**（CC 原生）；换 id 自然成新节点——**不需要我们自己锚定/编号**。
- **同一个 session-id 可能合法地出现在多个并列根/节点**：压缩续接有时会断开 `parentUuid` 链，同一会话的记录因此分成几段互不相连的链——builder 按"无父记录=根"如实拆成多个 `对话N`。这是**正常**、verify 也会通过；不是 bug。
- **节点身份 = 段首记录的 `### [时间戳]`**（builder 用它在重建时把人写记忆贴回，与路径/别名/记录数无关）。少数情况：一段对话内部**新分叉**导致原节点被切短、段首时间戳变了，旧人写记忆可能贴不回那一支——重建后留意该节点是否回到 `⚠️ 待补`，是就重贴一下。
- **思维导图.md** = Mermaid 流程图，GitHub 原生渲染、节点可点击跳到 `段.md`；随对话增多重跑即长大。

## 铁律

1. **生成物勿手改**：`段.md` / `TREE.md` / `思维导图.md` / `tree.json` / `规范.md` 都由脚本生成。要改→改源或脚本重跑。手改必漂移。
2. **只有 `研究历程.md` / `动机日志.md` 是人写的**节点记忆——脚本只脚手架、绝不覆盖。轻节点可短，但必须有实质内容、不能只留骨架。
3. **脱敏**：来自原始对话，私有仓 + 脚本脱敏报告 + **提交期 `check_pii.py` 闸**（扫漏脱敏的手机/邮箱/微信号/长号，含人写的研究历程/动机）+ 人工过目兜底（正则非 100%；命中 0 不等于绝对干净，含**报价/公司名/个人姓名**时正则抓不住，靠私有仓 + 人工判断）。
4. **提交精确 `add`**：共享 checkout，`git add -A` 会卷入别人 WIP；只 add 本人 `对话树/`。

## 节点记忆：摘要喂导图 · 历程 append-only

每个节点的 `研究历程.md` / `动机日志.md` 是**人写的节点记忆**，也是思维导图的内容来源：

- **`研究历程.md` 顶部必有一行「摘要：…」**——一句话讲清"这个分支做了什么"。**思维导图 / `TREE.md` 的节点标签就读这一行**（没填则回退自动截取原话）。所以：**先把节点的研究历程/动机写好，导图才"一看就知道做了什么"。**
- **摘要可迭代更新**（随认识深化改写）；**但历程正文（主线 / 转向 / 踩坑）只增不删（append-only）**——历史不抹。
- **`动机日志.md`**：每步**为什么** / 否决了什么。
- 顺序：**先写详情 → 摘要凝练 → 重跑 `build_session_tree.py` → 导图自动体现**。思维导图 = **全局概览 + 跳转目录**（看图知全貌、按 `TREE.md` 链接点进历史）。

### 任务目录（章内任务 + 可跳转）—— `研究历程.md` 必含

一个节点（一段对话）里会**陆续冒出很多任务**，所以光一句"摘要"不够。每个 `研究历程.md` 还要有一节 **`## 任务目录`**：把这个节点内部做的事，按层级列成一个**小目录**（像书的章→节→小节）。`build_session_tree.py` 把所有节点的任务目录汇成总 [`目录.md`](#)——**章按节点树嵌套，章内是该节点的任务；所有任务叶子从上往下读 = 整个项目的全部工作内容**。

格式（嵌套 markdown 列表，2 空格一级；**叶子任务末尾带 `[时间戳]`**）：

```markdown
## 任务目录

- 大任务A
  - 具体子任务 `[2026-06-15 07:04:26]`
  - 又一个子任务 `[2026-06-15 07:09:19]`
- 大任务B
  - 子任务 `[2026-06-15 07:27:58]`
    - 更细的小任务 `[2026-06-15 07:50:16]`
```

**为什么叶子带 `[时间戳]`**：这是让目录**可跳转**的锚点。`段.md` 里每一轮对话都有 `### [YYYY-MM-DD HH:MM:SS] 🧑/🤖` 标题；你给某个任务标上**驱动它那一轮的时间戳**（通常是发起该任务的那条用户消息），builder 就能据时间戳**自动算出它在 `段.md` 的当前行号**，在 `目录.md` 里生成跳转链 `段.md?plain=1#L<行号>`（GitHub 上点了直接跳到对话相应位置）。**时间戳是稳定锚点、行号每次重生成自动刷新**——所以你只管标时间戳、不用管行号。

怎么找时间戳：读该节点 `段.md`，找到驱动这个任务的那轮（多是 `🧑` 用户那条）的标题时间戳，原样抄进 `[…]`。

**append-only**：任务目录是历史——**只增不删**（做了新任务就往后加）；只有顶部「摘要」可随认识深化改写。

### 批量补全（一个子智能体补一个节点）
节点多时，按"**一个子智能体只分析一个节点**"分派：每个子智能体**只读该节点的 `段.md`**（那段真实对话）→ 写该节点的 `动机日志.md` + `研究历程.md`（含**摘要** + **任务目录**，任务带 `[时间戳]`）。互不干扰、可并行；补完重跑 builder，总目录与导图即刷新。

## 跨框架（Claude Code / Codex / Cursor）

> ✅ **现状：三家（Claude Code / Codex / Cursor）都已支持自动入树**（2026-07-05 落地）。builder 切成
> 「框架无关核心 + `scripts/adapters/` 每框架适配器」，用 `--adapters` 选源：

```bash
# 默认 = 全部已注册适配器（cc,codex,cursor）；也可显式限定
python3 .claude/skills/team-collab/scripts/build_session_tree.py --person <你> --adapters cc,codex,cursor
python3 .claude/skills/team-collab/scripts/verify_tree.py --person <你> --adapters cc,codex,cursor   # 校验要传同一集合
```

- **内部规范化格式 = CC 形状的记录 dict**：各适配器把本框架记录翻译成这形状（把 content 归一成
  text/tool_use/tool_result），于是 `entries_from_objs`/`render`/去重树对三家通用、无需各写一套。
- **Claude Code**：`~/.claude/projects/*.jsonl`，uuid/parentUuid 真实分叉树，marker+血脉发现。
- **Codex**：`~/.codex/sessions/**/rollout-*.jsonl`，`session_meta.cwd`（+全会话 `turn_context.cwd`）归属、子目录上卷；
  response_item 流映射；线性一会话一根；**Codex Desktop 导入的 CC 会话接成对应 CC 节点下 `分支（codex·…）`**
  （`external_agent_session_imports.json`），worker 子线程接成父 Codex 节点下分支；剥 IDE 注入噪声、截断超大工具输出、流式读。
- **Cursor**：`globalStorage/state.vscdb` 的 `composerData`/`bubbleId`（type 1=user/2=assistant），`workspace.json.folder`
  归属；线性一 composer 一根；只读打开（防运行时锁）、解析异常降级为空、不崩。
- **命名/回归**：CC 节点保持裸 `对话N（sid8）`（与旧产物字节一致）；Codex/Cursor 带 `codex·`/`cursor·` 前缀；
  tree.json 非 CC 节点带 `source_tool` 字段。设计/数据/测试细节见 skill 的 [`docs/`](../docs/README.md)。

> 加新框架：在 `scripts/adapters/` 加一个 `SourceAdapter`（discover/load/sid_of，产出 CC 形状 dict），
> 在 `adapters/__init__.py` 注册即可，核心无需动。见 [`conversation-log-spec.md`](./conversation-log-spec.md) 第四/五节。

## 为什么要有它

1. **组织记忆**：人走了、会话结束了，知识不流失。
2. **可继承的上下文**：新成员 / 新智能体读完，能"接上前任的脑子"。
3. **下一代智能体的成长语料**：`team-collab` 是"规范"（怎么做事），对话树是"经验"（事是怎么真正做成的）。

**可见性：团队所有人可见。** 这是组织记忆，透明共享。
