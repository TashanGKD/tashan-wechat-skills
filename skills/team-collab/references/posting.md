# 发帖 / 回帖详规（话题 → 帖 → 回复）

> 这是 `team-collab` 的**协作详规**，也是各智能体写协作 md 的**文档模板**。**只读本文件就能从零学会发帖**，不需要其他背景。
> 协作帖都在仓库的 `团队协作记录/` 目录里，每个文件是一个话题。

---

## ⚙️ 首次使用（每个 clone 各做一次）

帖子有**格式 gate**：提交时会自动校验格式，不合模板会被拦下。让本地提交也能即时校验，启用一次 git 钩子：

```bash
git config core.hooksPath .githooks
```

> 不设也能发帖——推送 / 开 PR 时 GitHub Action 会在服务端兜底校验；但建议设上，本地 `git commit` 就能当场发现格式问题、不用等 CI。
> 随时手动自检：`python3 .claude/skills/team-collab/scripts/check_posts.py`

---

## 🚀 速查：发一个帖就这 5 步

1. **找话题**：在 `团队协作记录/` 找到相关话题 md（没有就新建一个，见下「新建一个话题」）。
2. **发新帖 or 回帖**：要说一件新事 → 在话题文件**末尾加一个 `## 帖 N`**；要回应别人 → 在那个帖的「💬 回复」区**底部追加** `> [你的名字 · 日期]`。
3. **照模板填**：每个帖必含 发帖 / 状态 / ① 想做什么 / ② 方案·内容 / 💬 回复（模板见下）。**先说想做什么，再说方案。**
4. **更状态 + 索引**：改帖 / 话题状态，同步 `团队协作记录/README.md` 索引表。
5. **提交**：`git commit`（钩子校验格式，不合会拦你并指出哪错）→ 按 git 纪律推送（冲突就 `git fetch` + rebase，不强推）。

下面是每步的细节。

## 结构：一个话题 = 一个 md；话题下不断发帖、回帖

- **话题（topic）= 一个 md 文件**（`NNN-简短标题.md`），是一个版块 / 容器，代表一个大主题。
- **帖（post）= 话题文件里的一个 `## 帖 N` 区块**。一个话题下可以**不断发新帖**（每个帖是一件独立的事：一个构想、一次更新、一个征集、一个提问）。
- **回复（reply）= 在某个帖的「💬 回复」区追加**一条署名发言。
- 核心顺序不变：**先说「想做什么」，再说「方案」**。

> 别再把多件事揉进一个帖的流水账里——不同的事，在同一个话题下**各开一个帖**。

---

## 模板（照这个填，提交时按它过 gate）

```markdown
# 话题 NNN · <话题标题>

<!-- topic-meta
id: NNN
创建: YYYY-MM-DD
发起人: <名字>
状态: <🟢 进行中 | ⏳ 待回复 | ✅ 已结 | ⏸ 搁置>
摘要: <一句话：这个话题在讨论什么>
-->

> <话题说明：一两句，这个话题关于什么、谁该关注>

---

## 帖 N · <帖子标题>
- **发帖**：[<发帖人>] · YYYY-MM-DD
- **状态**：<🟢 进行中 | ⏳ 待 @某人 | ✅ 已结 | ⏸ 搁置>

**① 想做什么**
<动机 / 要解决的问题，先讲 Why>

**② 方案 / 内容**
<具体提议、要说明的内容、相关链接>

**💬 回复**
> [<回复人> · YYYY-MM-DD]
> <内容（append-only，不改别人已写的）>

**✅ 结论**（可选，拍板后写）
> 谁拍板 · 定了什么 · YYYY-MM-DD
```

## 新建一个话题

0. **先明确发起人**：谁发起、代表谁发起、谁推进。没明确就先问。
1. 在 `团队协作记录/` 下新建文件 `NNN-简短标题.md`，三位编号在现有最大值上 +1（先看 README 索引）。
2. 开头写 `# 话题 NNN · 标题` + `<!-- topic-meta -->`（id / 创建 / 发起人 / 状态 / 摘要）+ 一句话话题说明 + `---`。
3. 写第一个 `## 帖 1`（见下）。
4. **在 `团队协作记录/README.md` 的话题索引表追加一行**（话题级状态）。表头是 `| 编号 | 标题 | 发起 | 状态 |`，例如：
   ```
   | [008](./008-周报自动汇总.md) | 周报自动汇总 | [Alice] | ⏳ 待回复 |
   ```
5. **新帖文件 + README 索引改动要放进同一次提交**——gate 会检查"本话题已在 README 登记"，只提交新文件、不带 README 改动会被拦下。

## 在话题下发一个新帖

- 在话题文件**末尾追加**一个 `## 帖 N · 标题`（N = 现有最大帖号 +1）。
- 必含：`- **发帖**：[你代表的成员] · 日期`、`- **状态**：…`、`**① 想做什么**`（先讲动机）、`**② 方案 / 内容**`、`**💬 回复**`。
- 需要谁回应，状态标 `⏳ 待 @某人`。

## 回复一个帖

- **只在那个帖的「💬 回复」区底部追加**，格式：
  ```
  > [你代表的成员 · YYYY-MM-DD]
  > 你的发言……
  ```
- **append-only：绝不修改或删除别人已经写下的话。**
- 回复完按情况更新**那个帖的状态**（及话题头状态、README 索引），三处保持一致。
- 讨论出结论就在该帖写 `**✅ 结论**`（谁、定了什么、哪天），帖状态改 `✅ 已结`。

## 状态标签

🟢 进行中　⏳ 待回复（配 `@某人`）　✅ 已结　⏸ 搁置。话题头一个、每帖各一个；改了帖状态记得同步话题头 + README 索引。

状态值只要**包含**这四个图标之一即可、可带后缀——`⏳ 待 @XX78`、`🟢 进行中（结论已拍·余截图）` 都合法、都过 gate。

## 附件规则

- 仓库别处已有的（合同 `business/`、截图 `docs/screenshots/`）→ 相对路径链接，不复制。
- 只属于本次讨论的临时文件（草图、对比图）→ 放 `团队协作记录/附件/NNN/`，再在帖里链接。

## 一个话题里可以有几个帖？

按需。简单的事一个话题一个帖即可；一个长期主题（如某个 skill）会随时间长出多个帖（构想 → 更新 → 征集反馈 …）。判断标准：**一件能独立收敛出结论的事 = 一个帖**。

## 一个好回帖长什么样（示范）

假设你代表 `[XX78]`，要回复 001 话题里"请你 review 合同"的帖。**先读那个帖的 ① 想做什么搞懂动机**，再在它的「💬 回复」区底部追加：

```markdown
> [XX78 · 2026-06-16]
> 看完了，三点反馈：
> 1. 范围：合同第 4 条写"先行版不含万方实时检索"，但我已经在 integrations/wfdata-search 实现了关键词+详情+语义检索（没配 key 时返回 not_configured）。这块对不上，建议你定是纳入范围还是注明"预留接口"。
> 2. 验收第 7 项"论文正文生成"技术上可达，但"字数≥预算70%"在长论文下偶尔不稳，建议放宽到 60% 或按章计。
> 3. 其余 12 项验收没问题。
```

注意它：署 `[XX78]`、带日期、追加在 💬 回复区底部、给的是**技术事实和意见**（范围怎么改留给 Boyuan 定）、具体可操作。

## 🚩 红线：出现这些信号就停下、别动手

**总原则：违反规范的「字面」，就是违反规范的「精神」。** 别用"我没违反字面"给自己开脱。

下面这些借口，是真实压力下最常冒出来的（每条都来自对本 skill 的对抗测试）。出现任何一条，停下重想：

| 你正打算做 / 心里的借口 | 现实 |
|---|---|
| 以别人的名义写一句他没在仓库里写过的话（哪怕"他口头同意了"） | 伪造留痕。口头 / 转述 ≠ 本人落字。只能以**你自己的名**转述，并 @ 本人确认。 |
| 用自己的名替别人"作保"（写"XX 已确认…"，但他没真确认） | 比冒名更隐蔽的伪造。他没看过就是没看过，别替他背书。 |
| "甲方今天就要 / 时间太赶"，所以跳过确认、直接结案 | 时间压力不能把"没确认"变成"已确认"。赶出来的雷在验收期连本带利还。 |
| "这本来就是我的域，我直接拍了"——但帖子还等着对方的事实、或还有未决项 | 域内可拍，但缺关键事实 / 有未决子项时拍 = 蒙眼拍。先把事实和未决项收齐。 |
| "做都做了"（沉没成本），所以把它写进范围 / 合同 | 已经做了 ≠ 该纳入。纳不纳入是范围决定，按「找谁负责」走。 |
| "省事，几件事塞一个帖、别搞格式" | 一事一帖、模板是为了能各自收敛。塞一锅 = 没法分别拍板、互相串味。**短可以，混不行。** |
| 把还有实质未决项的 review 帖标 ✅ 已结 | ✅ 只给"该帖目的真的达成了"。还有 ⏳ 子项 / 未答问题 → 最多 🟢 进行中。 |
| 发起人还没明确，就先开帖或替别人开帖 | 先停。协作记录的第一责任人必须清楚：谁发起、代表谁发起、谁推进。没有明确就先问。 |

**自检一句话**：我是在「请对方提意见 / 如实留痕 / 在我域内基于充分事实拍板」，还是在「替别人拍他的板 / 替别人编一个确认 / 为赶时间抹掉未决项」？后者一律停。

## 格式 gate（提交会自动拦）

帖子提交时会过格式校验 [`../scripts/check_posts.py`](../scripts/check_posts.py)（pre-commit 钩子 + GitHub Action）：检查话题头 / 帖结构（发帖·状态·①·②·💬回复）/ 状态值 / 回复署名 / 编号一致 / README 登记。任一不合就**挡下提交并提示按本详规重写**。

- 本地启用钩子一次：`git config core.hooksPath .githooks`
- 手动自检：`python3 .claude/skills/team-collab/scripts/check_posts.py`
- 临时跳过（不建议）：`git commit --no-verify`
